## [블로그 개요]

- **주제:** Harness Engineering — AI 코딩 도구를 제대로 쓰는 방법
- **독자:** Vibe Coding을 시작해봤지만 "왜 잘 안 되지?" 싶은 개발자 및 비개발자
- **톤앤매너:** 친근하고 실용적. 어렵지 않게. 비유와 사례 위주. 강의하듯 쓰지 말고 경험을 공유하듯 써줘.
- **분량:** 2,500~3,500자 (읽는 데 10~15분)
- **목표:** 독자가 "아, 이게 문제였구나" 하고 무릎을 치고, 당장 써보고 싶어지는 글

---

## [글 구조]

### 도입부 (훅)
- Vibe Coding을 써봤는데 처음엔 신기하다가, 프로젝트가 조금만 커지면 AI가 이상한 짓을 하기 시작하는 경험으로 시작
- "코드를 잘 짜는 게 문제가 아니었다. AI에게 주는 맥락의 품질이 문제였다"로 연결
- 이 글에서 다룰 내용 한 줄로 예고

### 본론 1: Vibe Coding의 함정
- Vibe Coding이 뭔지 한 줄 정의 (자연어로 AI에게 지시해서 코드를 만드는 방식)
- 어떤 AI 코딩 도구든 (Claude, Codex, Cursor, Windsurf 등) 공통으로 겪는 3가지 문제를 구체적으로 묘사:
  1. **Context Contamination**: 대화가 길어질수록 AI가 앞에서 세운 규칙을 잊어버리기 시작함. "분명히 이 파일은 건드리지 말라고 했는데..." 상황.
  2. **원본 파일 사고**: AI가 원본 데이터를 덮어쓰거나 삭제해버리는 상황
  3. **인코딩·컬럼 충돌**: 한글 깨짐, 파일 버전이 바뀌면 컬럼명이 달라서 코드 전체가 터지는 상황
- "이 문제들의 공통 원인은 무엇인가?" 라는 질문으로 다음 섹션 연결

### 본론 2: Harness Engineering 개념 소개
- **비유로 시작**: "스타벅스에서 음료를 만들 때 바리스타 한 명이 주문, 제조, 계산, 청소를 혼자 다 하지 않는다. 역할이 나뉘어 있고, 각자 자기 파트만 한다. Harness도 똑같다."
- Harness의 정의: AI가 실수를 하지 못하도록 만드는 구조적 환경
- 4가지 핵심 요소 설명:
  1. **Context 파일** (rules 파일, PRD, ARCHITECTURE.md): AI가 매번 읽어야 할 규칙집. 직접 쓰지 않아도 된다 — AI가 대신 써준다.
  2. **Tool 경계**: 각 AI 에이전트가 접근할 수 있는 폴더와 기능을 명확히 제한
  3. **Hook (자동 안전장치)**: AI가 위험한 행동을 하려는 순간 자동으로 차단하는 장치. 코드로 구현되며 설계자 에이전트가 자동으로 설치해준다.
  4. **피드백 루프**: 검토자 에이전트가 규칙 위반을 리포트하면 rules 파일에 규칙을 추가하고 재실행

### 본론 3: GitHub harness-100 소개
- GitHub에 "harness-100"이라는 레포가 있다는 것을 자연스럽게 소개
- 독자가 검색할 수 있도록 "GitHub에서 harness-100 검색" 안내
- 레포의 내용: 다양한 프로젝트 유형에 맞는 rules 파일 템플릿, Hook 스크립트 100가지, PRD 템플릿 모음
- "처음부터 혼자 다 만들 필요 없다. 이미 잘 정리된 템플릿이 있다" 메시지

### 본론 4: 멀티 에이전트 구조 — "AI 팀을 구성하라"
- 핵심 패러다임 전환: AI 도구 하나에게 전부 시키는 것 → 역할이 나뉜 AI 팀
- 어떤 AI 코딩 도구를 쓰든 (Claude, Codex, Cursor, Windsurf, Cline 등) 이 구조는 동일하게 적용 가능하다고 명시
- 5개 에이전트 역할 테이블 (표로 정리):

| Agent | 별명 | 핵심 역할 | 모드 |
|-------|------|----------|------|
| Agent 1 | 설계자 | 규칙·구조·Hook 설계 | 계획만 (파일 수정 없음) |
| Agent 2 | 수집가 | 데이터 읽기·저장 | 실행 |
| Agent 3 | 분석가 | 계산·분석 | 실행 |
| Agent 4 | 개발자 | 화면·UI 구현 | 실행 |
| Agent 5 | 검토자 | 규칙 준수 감사 | 계획만 (파일 수정 없음) |

- **왜 새 대화를 열어야 하는가**: Context Contamination 문제. 대화가 길어지면 AI가 규칙을 잊는다. 역할이 바뀔 때마다 새 AI 대화를 시작해야 한다.
- **계획 모드 vs 실행 모드** 개념 설명: 많은 AI 코딩 도구가 이 두 모드를 지원한다. 계획 모드 = 파일 건드리지 않고 설계만 / 실행 모드 = 실제 코드 작성. 지원하지 않는 도구라면 프롬프트에 "계획만 세워줘. 코드나 파일은 만들지 마"를 명시하면 된다.

### 마무리
- 핵심 교훈 3줄 요약:
  1. 직접 규칙 파일 쓰지 마라 — 설계자 에이전트가 써준다
  2. 역할을 나눠라 — AI 하나가 전부 다 하면 무너진다
  3. 새 역할 = 새 대화 — Context Contamination 차단의 핵심
- "2편에서는 실전 프로젝트에 이것을 어떻게 적용하는지, 복붙만 해도 되는 프롬프트와 함께 공유한다"로 예고

# ✍️ 2편 프롬프트: "Harness Engineering 실전 적용"

## [블로그 개요]

- **주제:** Harness Engineering 실전 — 복붙 프롬프트로 AI 팀 만들기
- **독자:** 1편을 읽고 "그래서 실제로 어떻게 하는 건데?" 가 궁금한 독자
- **톤앤매너:** 튜토리얼 스타일. 단계별로 따라올 수 있게. 실제로 복사해서 쓸 수 있는 프롬프트 제공.
- **분량:** 3,000~4,000자 (읽는 데 15~20분)
- **목표:** 독자가 이 글만 보고도 자신의 프로젝트에 당장 적용할 수 있어야 한다

## [글 구조]

### 도입부
- 1편 내용 한 줄 요약으로 시작 ("지난 글에서 Harness가 무엇인지 살펴봤다. 오늘은 실제로 적용해보자.")
- 오늘 만들 것 소개: 레거시 데이터 파일(CSV/Excel)을 읽어서 분석하고 시각화하는 Tool
- **예시 프로젝트 설정** (회사 특정 없이 일반적으로):
  - 시나리오: 어딘가에 오래된 레거시 시스템이 있고, 데이터는 파일로만 추출 가능 (직접 연결 불가)
  - 목표: 그 파일들을 자동으로 읽어서 트렌드와 이상 징후를 시각화하는 대시보드 Tool

### 본론 1: 프로젝트 시작 전 — 제약 조건 3가지 정의

먼저 독자가 종이에 적어야 할 3가지:
1. **데이터 제약**: 원본 파일을 건드릴 수 있는가? (대부분 No)
2. **인코딩**: 한글이 있는가? 어떤 인코딩인가?
3. **파일 구조**: 컬럼명이 고정인가, 버전마다 바뀌는가?

이 3가지가 rules 파일(AI 규칙집)의 핵심 내용이 된다고 설명.

### 본론 2: Step 1 — 설계자 에이전트 실행

- **실행 방법**: 사용 중인 AI 코딩 도구에서 새 대화를 열고, 계획 모드가 있다면 ON으로 설정. 없다면 프롬프트 첫 줄에 "계획만 세워줘. 코드나 파일은 절대 만들지 마."를 추가.
- 아래 프롬프트를 통째로 복붙하라고 안내:

```
[설계자 에이전트 프롬프트 — 복붙용]

너는 수석 소프트웨어 아키텍트야.
계획만 세워줘. 코드나 파일은 절대 만들지 마.

[프로젝트 설명]
목적: [데이터를 분석해서 트렌드와 이상 징후를 시각화하는 대시보드 Tool]
데이터 소스: [CSV/Excel 파일 (레거시 시스템에서 파일로만 추출 가능)]
제약사항:
- 원본 파일 수정·삭제 불가
- 데이터에 한글 포함 가능 (인코딩 처리 필수)
- 컬럼명이 파일 버전마다 달라질 수 있음 (하드코딩 금지)

아래를 순서대로 설계해줘:
1. AI가 지켜야 할 규칙 (rules 파일 초안)
   - data/raw/ 원본 파일 보호 규칙
   - 인코딩 필수 명시 규칙
   - 컬럼명 하드코딩 금지 규칙
2. 만들 것 / 절대 만들지 않을 것 (PRD)
3. 폴더 구조 설계
4. Hook 3가지 설계
   - 원본 파일 쓰기 자동 차단
   - 인코딩 미지정 자동 차단
   - 컬럼명 하드코딩 감지
5. Agent 2~5 역할 정의서
```

- 설계자가 결과를 보여주면: 검토 후 실행 모드로 전환 → 파일 생성 승인
- Hook 설치 확인 방법 설명 (rules 파일이 프로젝트 루트에 생성됐는지 확인)

### 본론 3: Hook 3가지 — 자동 안전장치 작동 방식

각 Hook에 대해 Before/After 코드 예시와 함께 설명:

**① Raw File Guard**
- Before (AI가 하려던 것): `df.to_csv("data/raw/원본파일.csv")`
- Hook 작동 → After: `df.to_csv("data/processed/결과파일.csv")`
- "이게 없으면 몇 달치 원본 데이터가 AI 실수 하나로 날아간다"

**② Encoding Guard**
- Before (AI가 하려던 것): `pd.read_csv("data/raw/파일.csv")` (인코딩 없음)
- Hook 작동 → After: 인코딩 자동 감지 후 `pd.read_csv(path, encoding='euc-kr')`
- "한글 데이터에서 인코딩 빠뜨리면 모든 한글이 ??? 로 나온다"

**③ Hardcode Guard**
- Before (AI가 하려던 것): `df["매출금액"]` (컬럼명 직접 입력)
- Hook 작동 → After: `df[schema["revenue"]]` (스키마에서 참조)
- "파일 버전 하나 바뀌면 컬럼명이 달라져서 전체 코드가 터진다"

### 본론 5: Step 5 — 검토자 에이전트 (마지막 안전망)

- 계획 모드 또는 "코드 수정하지 말고 리포트만 써줘" 명시 후 실행
- 검토 항목 5가지를 체크리스트 형식으로 나열:

  2. 인코딩 미지정 read_csv가 있는가?
